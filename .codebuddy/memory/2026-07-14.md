## 前端打包体积优化（workouts_log dashboard）

- **根因**：`src/App.tsx` 用 `import rawActivities from './static/activities.json'` 把 ~1.9MB 运动数据**静态打进了 `index.js`**，导致主包 2.07MB（gzip 617KB）。recharts 实际未被打包（图表组件 MonthlyChart/TrendCharts 未真正渲染），之前误以为是 recharts 占体积。
- **最终方案（重要，避免重蹈覆辙）**：fetch 方案会让应用**额外依赖一次成功的运行时网络请求**才能渲染；在 `file://` 直接打开、静态托管对 `.json` 返回 HTML 404、或带 `PATH_PREFIX` 子路径解析偏差等环境下会失败（表现为无法访问/卡加载）。已**回退 fetch**，改为：数据仍静态 `import`，但用 `manualChunks` 规则 `if (id.includes('activities.json')) return 'data'` 把 JSON 单独拆成 `data-Ik62Ct3x.js`（~1.98MB）chunk。`index.js` 仍为 **86.6KB**（gzip 25KB），运行时模型与改动前完全一致（无需 fetch，任何部署方式都能访问）。
- **manualChunks 健壮性**：`vite.config.ts` 的 manualChunks 已兼容 pnpm 的 `.pnpm/<pkg>@<ver>/node_modules/<pkg>/` 布局（用包名子串匹配而非 `/包名/`），并按 data(activities.json) / mapbox / charts(recharts,d3) / icons(lucide) / react-vendor / vendor 拆分。`chunkSizeWarningLimit` 调至 2000。
- **剩余较大块**（非阻塞）：`mapbox` 1.78MB（首页 RouteMap 被静态引入，可改为 lazy 懒加载进一步延迟）、`china-provinces` 577KB（中国地图 geojson，已拆分）。
- 调试技巧：在 manualChunks 内 `console.error(id)` 可打印模块真实路径（pnpm 用正斜杠，形如 `node_modules/.pnpm/mapbox-gl@3.23.1/node_modules/mapbox-gl/...`）。

## ActivityLog.tsx 固定模块高度（每页 15 行）

- 需求：`ActivityLog` 每页最多 15 条（`PAGE_SIZE = 15`），且模块总高度恒定，不随记录数量/名称长短变化；空列表也保留翻页区（按钮置灰禁用）。适用于 全部/跑步/骑行/徒步/健身 五页。
- 实现：tbody 始终渲染 15 行——真实数据不足用占位空行 `filler` 补齐（`Array.from({length: Math.max(0, PAGE_SIZE - pageData.length)})`）。占位行 td 用 `border-transparent`（无可见下框线但保留 1px 行高）。翻页区去掉 `sorted.length>0` 包裹恒显示，`totalPages = Math.max(1, ...)`，按钮靠 `disabled` 自然置灰。
- **高度不一致的坑（关键）**：数据行含 `TypePill`（inline-flex 胶囊 `py-0.5`+align-middle）被撑到 **45px**，而占位行只有单行文本 `&nbsp;`（`py-3`=44px）。`<tr>` 上的 `height` 只是**最小高度**，导致数据行 45 / 占位行 44 不等，总高随两类行占比变化。
- **最终可靠解法（固定高度容器）**：靠「强制 `<td>`/`<tr>` 高度」在用户的 Chrome+Firefox 上始终无效（build/dev 都复现），无论用作用域 CSS、内联 `style={{height,overflow}}` 还是 `overflow:hidden` 都不生效——表格布局算法使 height 仅为最小高度且内容会撑高行。
  - 改法 v2（消除滚动条）：上一版 `height: theadH + 15×45` 用「假设值」写死，表格真实渲染高度（边框 + 亚像素取整）略大 → `overflow:auto` 溢出出现右侧滚动条。改为**实测整表真实自然高度**锁定容器：`useLayoutEffect` 量 `innerRef`（包裹 `<table>` 的内层 div）的 `getBoundingClientRect().height`，容器 `style={{ height: tableH, overflow: 'hidden' }}`。高度精确匹配内容 → 无滚动条、也不裁切记录；字体加载后 `document.fonts.ready` 重测、resize 重测。
  - **真正的根因（关键！）**：固定高度容器 + 实测高度仍随年份跳动，因为**占位行是单 `colSpan` 单元格，而数据行是多单元格**——在 `border-collapse: collapse`（Tailwind Preflight 默认）下，这两类行在同样 `height:45` 下真实高度不一致（单跨列单元格略矮于多单元格行）。于是「数据行数/占位行数」配比随年份变化 → 总高变化。翻页看似稳定只是因为 `page` 不在测量依赖、沿用首屏旧值（假象）。
  - **最终修复（v3 · line-height 锁高）**：占位行结构与数据行已一致仍跳动 → 根因是 `height:45` 在表格里只是**最小高度**，内容（TypePill inline-flex / 文本基线）会把数据行撑过 45px，空占位行撑不到 → 两类行不等高。
    - 在 `src/index.css` 的 `.activity-log-table tbody td` 改用 **`line-height: 45px; padding:0; white-space:nowrap; overflow:hidden; box-sizing:border-box; vertical-align:middle`**（保留 `height:45px`）。`line-height` 直接决定单行文本盒高度（非「最小高度」），无论单元格内容（emoji 胶囊/长名称）如何，每格精确 45px → 数据行与占位行天然相等、总高恒定。
    - 容器仍用 `useLayoutEffect` 实测 `innerRef` 整表高度锁定（`height:tableH; overflow:hidden`），字体加载/resize/`page` 变化重测；因行已统一，测得值恒定 → 无滚动条、不裁切、高度稳定。
    - 数据/占位单元格原本的 `py-3` 被该 CSS 的 `padding:0` 以更高优先级覆盖（`.activity-log-table tbody td` 特异性高于 `.py-3`），无需逐个改 className。
- **【2026-07-14 晚 · 已回退】用户最终决定放弃固定 ActivityLog 模块高度**：改了多轮（作用域 CSS → 内联 height+overflow → 固定高度容器+实测 → line-height 锁高）仍无法在 Chrome/Firefox 稳定消除「模块高度随记录数/年份跳动」，且伴随滚动条等问题。用户要求退回到 18:07 首次修改前的初始版本（保留其原有的 `h-[45px]` 行高与单 colSpan 占位行逻辑），不再固定模块高度。已手动还原 `ActivityLog.tsx` 与 `index.css`（环境无 git，无法 git revert）。**后续如用户再提类似需求，先确认是否仍要固定高度，避免重走一遍。**

  - 作为辅助（让行高尽量均匀、避免滚动条）：数据行所有 `<td>` + 占位行 `<td>` 内联 `style={{ height: 45, overflow: 'hidden' }}`；`TypePill` 锁 `h-[20px] leading-none overflow-hidden items-center`（去 `py-0.5`）。`<tr>` 保留 `h-[45px]` 作 min-height。
  - 三者（占位行补齐 15 行 + 单元格锁高 + 容器固定高度）共同保证模块恒定高度、空行无框线、空/不足一页翻页区保留且按钮禁用。
  - 真正坑点：**Safari 对 `<td>` 设 `overflow:hidden` 不会强制行高**（已知 bug，单元格所需高度仍含溢出内容，emoji 胶囊会把数据行撑高）。所以「靠 `<td>` 强制高度」在所有浏览器都不可靠；正确做法是**不让内容有机会撑高行**——把胶囊锁死固定高度，行高自然统一。
  - 历史尝试（均无效/撤回）：① 仅 `<tr>` 上 `h-[44px]/h-[45px]`（height 只是 min，emoji 撑高数据行）；② 外部 CSS `.activity-log-table tbody td { height; overflow:hidden }`（Safari 不强制 td 高度 / 环境可能未加载）；③ 全 `<td>` 内联 `height+overflow`（Safari 仍不强制）。最终靠「约束 TypePill 内容高度」解决，跨浏览器可靠。
  - 名称列已 `truncate` 单行防换行；`tbody` 恒 15 行由占位行 `filler` 补齐（`Array.from({length: Math.max(0, PAGE_SIZE - pageData.length)}`)）。
